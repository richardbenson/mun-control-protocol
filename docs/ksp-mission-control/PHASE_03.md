# Phase 3 — Career foundation + `get_tech_tree`

## Summary
Stand up the `KSPMissionControl.Career` kRPC service extension, establish the threading-safe state cache pattern that every later Career service will reuse, ship the first real service (`TechTreeService`), run `krpc-clientgen` for the first time, commit the generated stubs, and expose `get_tech_tree` on the MCP server end-to-end.

## Context
This is the highest-risk phase in the project. It introduces:
- A second build artefact (`KSPMissionControl.Career.dll`) that lives in KSP's `GameData/`
- A Unity-thread-safety constraint that, if broken, causes intermittent NullReferenceExceptions
- An external code-generation step (`krpc-clientgen`) that requires a running game

Every later phase that adds a Career service follows this phase's pattern. **The code style and threading approach established here is load-bearing.** Take the time to do it right.

The threading rule, restated from the requirements doc: KSP API calls must run on Unity's main thread. kRPC service procedures execute on kRPC's own thread. Therefore Career services serve from a **cache** that is repopulated on `MonoBehaviour.Update()` (or scene-change events for things that don't change frame-to-frame). Tech tree state changes only on R&D actions, so a less-frequent refresh trigger is fine.

## Files expected to change
- `src/KSPMissionControl.Career/KSPMissionControlAddon.cs` — new (the `MonoBehaviour` entry point that registers services and pumps the cache)
- `src/KSPMissionControl.Career/Services/TechTreeService.cs` — new (the kRPC service)
- `src/KSPMissionControl.Career/Internal/StateCache.cs` — new (the threading-safe cache pattern, reusable)
- `src/KSPMissionControl.Shared/Models/TechNode.cs` — new
- `src/KSPMissionControl.MCP/Krpc/KSPMissionControlStubs.cs` — generated by `krpc-clientgen`, committed
- `src/KSPMissionControl.MCP/Tools/CareerTools.cs` — extend with `GetTechTreeAsync()`
- `deploy/build-and-deploy.ps1` — new helper script (PowerShell) that builds the Career project and copies the DLL into the KSP `GameData/KSPMissionControl/` folder

## Acceptance criteria
1. `dotnet build` produces `KSPMissionControl.Career.dll`; the deploy script copies it to `$KspInstallDir/GameData/KSPMissionControl/`.
2. With KSP launched against a career save and the addon loaded, `krpc-clientgen csharp KSPMissionControl ...` produces a stubs file. The file is checked in.
3. `get_tech_tree` MCP tool returns a non-empty list of tech nodes whose statuses (unlocked / available / locked) match what's visible in-game in the R&D tree.
4. The MCP tool's response includes, per node: node id, title, science cost, status, and the list of part names in that node.
5. `dotnet test` still passes (a new test asserts `TechNode` JSON round-trip; the new MCP tool gets a unit test mocking the kRPC stub).
6. PR opened against `feature/ksp-mission-control` from `feature/ksp-mission-control-phase-03`.
